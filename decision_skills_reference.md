# Decision Skills — Reference

Every kind of idea I work on — AIMH MVP, hackathon pitch, team workflow experiment, career move — is the same skill applied to a different target. The default is "don't build it." The burden of proof is on the idea to earn the next stage.

---

## The Six Parts

### 1. The Diff
What does the target user's life look like before this exists, and what does it look like after? Stated in one sentence. If I can't show a clean before/after, I don't have an idea yet — I have a topic.

### 2. The Risky Assumption
Of all the things that have to be true for this to matter, which am I least sure about? Usually it's "does anyone care," not "can I build it." Whichever it is, that's what gets tested first.

### 3. The Cheapest Test
The smallest, fastest, lowest-effort action that would actually move my confidence in the risky assumption. Not "build the MVP." A spreadsheet, five conversations, a landing page, a manual mockup. Building is almost never the cheapest test.

### 4. The Kill Criteria
Written *before* the test runs. "I continue if [specific outcome] within [specific window]." No criteria = the idea never dies, it just drifts into "parked," which is a graveyard with a nicer name.

### 5. The Verdict
Either the criteria were met (continue) or they weren't (archive). Killing counts as shipping. Parked is not a real status.

### 6. The Artifact
Every idea produces something public or shareable, regardless of verdict. AIMH: a video. Corporate: a writeup. Career: a note to future me. The artifact makes the verdict real and prevents revisionist memory.

## Applied: AIMH MVP

- **Diff:** before = user does X in a clunky way; after = user does X in a clearly better way on one screen.
- **User:** a specific named person or tiny segment, not "people."
- **Risky assumption:** usually "anyone besides me wants this."
- **Cheapest test:** show 5 target users a manual version. Look for "when can I have it" or "can you text me when it's ready."
- **Kill criteria example:** 3 of 5 ask to be notified within 2 weeks.
- **Artifact:** YouTube episode showing the gauntlet, regardless of verdict.

## Applied: Hackathon Pitch

- **Diff:** existing product feature → small visible improvement. Borrow the audience's existing mental model so I don't have to bootstrap a world.
- **User:** the judge in the room. Three minutes, zero setup time.
- **Risky assumption:** can I show the change without explaining the world it sits in?
- **Cheapest test:** write the pitch first. Record it out loud. If it doesn't land in 3 minutes, scope is wrong.
- **Kill criteria:** the pitch fits in 3 minutes with a clean before/after.
- **Artifact:** the pitch and recording.

## Applied: Corporate Experiment (pen test team)

- **Diff:** one repeated workflow → one specific change for one week. Triage, reporting, recon, scoping — pick one.
- **User:** the team. Including me.
- **Risky assumption:** does this change actually make the work better, not just newer?
- **Cheapest test:** one person tries the change for one week, reports back in a shared doc.
- **Kill criteria:** explicit better/worse/neutral verdict at end of week. Worse is fine — still shipped.
- **Cadence:** weekly, one experiment per person. Not daily. Daily becomes theater.
- **Artifact:** internal writeup. The accumulating doc becomes the team's experimentation record.

## Applied: Career Decision

- **Diff:** what does my week actually look like before vs. after this change? Not the title, not the company — the actual lived hours.
- **Risky assumption:** does this move me toward "build and create" or just upgrade the surroundings of "orchestrate"?
- **Cheapest test:** a conversation with someone currently in that role, or a one-day shadow, or running the framework against the job description honestly.
- **Kill criteria:** written before the offer is real. "I take this if it includes X and Y. I don't take this if it's primarily Z."
- **Artifact:** a note explaining the decision to future me, including what would make me reverse it.

## Anti-Patterns

- **Building before validating.** If the test involves writing code, the test is probably wrong.
- **"Parked."** Not a real status. Park = archive with a nicer name. An idea is either alive with active kill criteria or it's archived.
- **Premature instrumentation.** Analytics, dashboards, trackers — these earn their place after something shows life, not before.
- **Tool-building hijacking the work.** When I'm improving the tracker instead of testing the ideas, the meta-project has eaten the object-level project. Stop. The tracker should be so boring I forget it exists.
- **Sunk-cost dressed as efficiency.** "I already built X, so I'll use X for this new job" — different jobs, evaluate independently.
- **Idea → prototype → deck in that order.** Wrong order. Deck (or its plain-text equivalent) first, because it's the cheapest test of "does anyone understand and care." Prototype only after the cheap tests pass.
- **No verdict.** The work isn't done when the test ends. It's done when there's a written verdict and an artifact.
- **Apply the framework everywhere except to my own tools.** When I find myself proposing to build or simplify a tool, that's itself an idea. Run it through the gauntlet.
